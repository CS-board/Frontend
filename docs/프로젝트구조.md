# 폴더구조 / 인수인계용

백엔드 API는 `NEXT_PUBLIC_API_BASE_URL`(예: `…/api`) 기준으로 `api/client.ts`가 호출합니다.

## 디렉터리 역할

| 경로 | 설명 |
|------|------|
| `app/` | Next.js App Router. URL = 폴더 구조. 각 `page.tsx`가 화면 진입점. |
| `app/*/page.tsx` | 라우트별 페이지(대부분 `"use client"`). |
| `components/features/` | 도메인용 UI: 사이드바, 홈(데스크톱/모바일), 푸터 등. |
| `components/ui/` | shadcn/ui 기반 공통 컴포넌트(버튼, 카드, 입력 등). |
| `services/` | API 호출만 담당. 페이지에서 직접 `fetch` 하지 않고 여기 모음. |
| `api/client.ts` | 공통 HTTP 클라이언트. JWT(AT) 첨부·만료 시 갱신·401 재시도. |
| `constants/index.ts` | `ROUTES`, `API_ENDPOINTS`, `TOKEN_KEY`, 반응형 브레이크포인트. |
| `types/index.ts` | 백엔드 응답 형태에 맞춘 TypeScript 타입. |
| `hooks/` | `use-auth`(로그인 상태), `use-mobile`, `use-toast` 등. |
| `lib/utils.ts` | `cn()` — Tailwind 클래스 병합. |
| `public/` | 정적 자산(로고 등). |

## 데이터 흐름(요약)

1. 사용자 액션 → `services/*.ts` → `apiClient.get/post/patch/delete`
2. `apiClient`가 `localStorage`의 AT를 헤더에 넣고, 필요 시 `/auth/refresh`(쿠키 RT)로 갱신
3. 응답은 `{ success, data }` 형태를 가정하고 `services`에서 `res.data`만 꺼냄

### 개선 여지

| 항목 | 제안 |
|------|------|
| `services/ranking.ts` | 현재 앱은 `challengeService` 위주. 미사용이면 삭제하거나 README에 “레거시” 명시. |
| 홈 목업 | `weekly-ranking.tsx`, `challenge-info.tsx` 등 일부는 샘플 데이터 — 실API 연동 여부를 문서에 적어두면 혼동 감소. |
| 환경 변수 | `.env.example`에 `NEXT_PUBLIC_API_BASE_URL`만 있어도 후임이 바로 실행 가능. |
| 테스트 | 단위/E2E가 없으면 포폴에서는 “추가 예정” 또는 수동 체크리스트를 README에 한 줄. |
| `app/` 비대화 | 페이지가 커지면 `features` 쪽으로 커스텀 훅·하위 컴포넌트 추출을 권장. |
